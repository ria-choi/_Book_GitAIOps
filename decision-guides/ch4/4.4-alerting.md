# 4.4 알림 방식 선택

## 상황

4.2에서 Prometheus와 Grafana를 설치하고, 4.3에서 Loki로 로그를 수집하고 있다. 메트릭과 로그를 볼 수는 있지만, 대시보드를 24시간 지켜볼 수는 없다. 문제가 생기면 자동으로 알려주는 알림 시스템이 필요하다.

## 독자가 물어볼 수 있는 질문

- "알림은 어떻게 설정해?"
- "Prometheus 알림이랑 Grafana 알림이 뭐가 달라?"
- "Slack이나 PagerDuty로 알림 보낼 수 있어?"
- "어떤 조건에서 알림을 보내야 해?"

## 추천: Grafana Alerting

Grafana 자체의 Alert Rule 기능으로 알림을 설정한다. Prometheus 데이터소스를 쿼리하여 조건을 평가하고, Contact Point(Slack, Email 등)로 알림을 보낸다.

이 프로젝트에 적합한 이유:
- **이미 설치됨**: 4.2에서 Grafana가 이미 동작 중 → 추가 설치 불필요
- **UI 기반**: Alert Rule을 Grafana UI 또는 provisioning YAML로 설정
- **통합 경험**: 대시보드에서 메트릭 확인 → 같은 도구에서 알림 설정 → 일관된 워크플로우
- **다양한 채널**: Slack, Email, PagerDuty, Webhook 등 Contact Point 지원

## 비교

| 방식 | 특징 | 장점 | 단점 | Notiflex 적합도 |
|------|------|------|------|----------------|
| **Grafana Alerting** | Grafana 내장 알림 | 추가 설치 없음, UI 직관적, 다양한 채널 | Grafana 의존, 대규모에서 Alertmanager보다 약함 | ★★★ |
| **PrometheusRule + Alertmanager** | Prometheus 스택 네이티브 | 강력한 라우팅/그루핑, GitOps 호환(CRD) | Alertmanager 별도 설정, 학습 곡선 | ★★ |
| **PagerDuty / OpsGenie** | 전문 인시던트 관리 | 에스컬레이션, 온콜 스케줄, 모바일 앱 | 유료, 학습 환경에서 과도 | ★ |
| **Cloud Monitoring Alert** | GCP 네이티브 | GCP 콘솔 통합, SMS 지원 | Prometheus 메트릭 직접 못 씀, 별도 설정 | ★ |

> **Notiflex 맥락**: Grafana가 이미 설치되어 있고, 학습 환경에서 알림의 핵심 개념(조건, 평가 주기, 알림 채널)을 경험하는 것이 목적이다. PrometheusRule + Alertmanager는 프로덕션에서 더 강력하지만 설정이 복잡하다. Grafana Alerting으로 개념을 익히면 Alertmanager 전환은 어렵지 않다.

## 핵심 개념

- **Alert Rule**: "이 조건이 N분 동안 지속되면 알림을 발생시켜라"는 규칙. 예: `kube_pod_container_status_restarts_total > 0`이 5분간 지속.
- **Contact Point**: 알림을 받을 대상. Slack webhook, 이메일, PagerDuty 등을 설정한다.
- **Evaluation Interval**: Alert Rule을 얼마나 자주 평가할지. 1분이면 매분 조건을 확인한다.
- **Notification Policy**: 어떤 알림을 어떤 Contact Point로 보낼지 라우팅 규칙. 심각도별로 다른 채널에 보낼 수 있다.
