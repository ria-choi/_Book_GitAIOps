# 5.4 아키텍처 결정 기록(ADR)

## 사전 조건
- ch3·ch4에서 결정한 도구가 모두 적용된 상태 (ArgoCD, GitHub Actions, Prometheus + Grafana, Loki + Fluent Bit, PrometheusRule + Alertmanager)
- ch5.2 Gateway API + ch5.3 Blue/Green 적용 완료
- `notiflex-platform/docs/` 디렉터리 존재

## 실행 지침

ADR(Architecture Decision Records)는 결정의 "왜"를 git 저장소 안 마크다운에 누적하는 패턴이다. memory가 머신 로컬에 쌓이는 **나의 작업 컨텍스트**라면, ADR은 git에 누적되는 **팀의 결정 기록**이다.

### 단계 1: docs/architecture-decisions.md 신설

`notiflex-platform/docs/architecture-decisions.md`를 새로 만든다. 한 결정에 한 항목, 시간 순서대로 누적한다. 이전 장의 결정도 같이 정리한다.

머리 형식:
```markdown
# Architecture Decision Records
```

### 단계 2: ADR 항목 형식

각 ADR은 같은 형식을 따른다. 형식이 일관되어야 시간이 지나도 빠르게 훑을 수 있고 새 팀원이 패턴을 학습한다.

```markdown
## ADR-NNN: 짧은 결정 제목 (N장)
**시점**: YYYY-MM / **결정**: 한 문장으로 핵심 결정. 무엇을 채택하고 무엇을 안 쓰는지 명시
**이유**:
- 이유 1 (가장 결정적인 근거)
- 이유 2
- 이유 3
- 이유 4 (필요 시)
```

### 단계 3: 이전 장 결정 정리 (3장·4장)

이전 장 결정도 시간 순서로 누적한다. 본문 시연에서는 5개 항목 정리.

| ADR | 결정 | 출처 |
|---|---|---|
| ADR-001 | 배포 자동화는 ArgoCD | 3장 (decision-guides/ch3/3.2-gitops-tool.md) |
| ADR-002 | CI 도구는 GitHub Actions | 3장 (decision-guides/ch3/3.4-ci-tool.md) |
| ADR-003 | 메트릭은 Prometheus와 Grafana | 4장 (decision-guides/ch4/4.2-metrics-monitoring.md) |
| ADR-004 | 로그는 Loki와 Fluent Bit | 4장 (decision-guides/ch4/4.3-logging.md) |
| ADR-005 | 알림은 PrometheusRule과 Alertmanager | 4장 (decision-guides/ch4/4.4-alerting.md) |

각 항목의 "이유"는 decision-guide의 "이 프로젝트에 적합한 이유" 섹션에서 옮겨 적는다 (가드레일·decision-guide·ADR이 같은 근거를 공유하므로 일관성 확보).

### 단계 4: 5장 결정 추가 (ADR-006, ADR-007)

이번 장에서 한 두 가지 결정을 추가한다.

```markdown
## ADR-006: 외부 진입점은 Gateway API (5장)
**시점**: YYYY-MM / **결정**: GKE Gateway API로 외부 진입점을 만든다. Ingress는 사용하지 않는다.
**이유**:
- GKE 네이티브 지원: 별도 Controller 설치 불필요
- 리소스 효율: NGINX Ingress는 100m/90Mi 추가, Gateway API는 0
- Blue/Green 연동: HTTPRoute의 backendRefs weight로 트래픽 분배
- K8s 공식 표준: Ingress의 후속 세대

## ADR-007: 무중단 배포는 Blue/Green (5장)
**시점**: YYYY-MM / **결정**: Argo Rollouts의 Blue/Green 전략을 사용한다. Canary는 도입하지 않는다.
**이유**:
- Rolling Update 한계: 배포와 검증이 분리되지 않음
- 2 replica 규모에서 리소스 2배 부담이 크지 않음
- Canary는 메트릭 기반 자동 판정이 선행되어야 안전 (현재 미확보)
```

### 단계 5: 도구 변경 의견 요청 시 ADR 참조

독자가 "Ingress가 더 친숙한데 바꾸는 게 나을지 의견만 알려줘"처럼 도구 변경 가능성을 물으면, **`docs/architecture-decisions.md`의 해당 ADR을 직접 읽고 결정 이유를 근거로 의견**을 답한다. 머신 로컬 memory가 아니라 git 저장소의 영구 기록을 근거로 답하므로 다른 머신·다른 시점에서도 같은 답이 나온다.

답변 골자:
1. ADR을 읽었다고 명시
2. 결정 이유 3~4가지를 그대로 인용
3. 변경 시 영향(Controller 설치, 매니페스트 재구성 등) 명시
4. 권장 여부 + 변경 이유를 되묻기

## 트러블슈팅

### ADR 번호 충돌
→ 새 ADR을 추가할 때 기존 파일에서 가장 큰 번호를 확인 후 +1로 매긴다. 시간 순서가 깨지면 회고에서 결정 흐름을 추적하기 어려워진다.

### 결정 이유가 한 줄로 끝남
→ 4가지 이유 골격(GKE 네이티브 / 리소스 효율 / 연동성 / K8s 표준 같은)을 따라 다각도로 정리한다. 한 줄짜리 이유는 시간 지나면 컨텍스트가 사라진다.

### 같은 결정이 ADR과 decision-guide에 중복
→ 중복이 정상이다. decision-guide는 **고를 때** 참조하는 비교표, ADR은 **고른 후 누적**되는 결정 기록. 두 위치는 서로 다른 시점·다른 사용자(AI vs 사람)를 위한 자료다.
